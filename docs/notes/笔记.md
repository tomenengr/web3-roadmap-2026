# 📔 Web3 Roadmap 2026 学习笔记

## 📅 3.28 | Solidity 进阶语法与 Foundry 核心

### 1. 函数可见性与传参优化
- **calldata 优化**：Solidity 0.6.9 之后，`public` 函数的参数也可以指定为 `calldata` 以节省 Gas（避免从 calldata 拷贝到 memory）。
- **external 的限制**：
    - 定义为 `external` 的函数在合约内部**不能直接通过函数名调用**。
    - **强制调用方式**：必须使用 `this.funcName()`。
    - **注意**：`this.funcName()` 会触发一次真实的 EVM `CALL` 指令，相比内部跳转（JUMP）更加昂贵。

### 2. 错误处理与数学安全 (Solidity 0.8.x+)
- **Gas 优化 (if-revert)**：0.8.4 之后，推荐用 `if (!condition) revert CustomError()` 替代 `require`。自定义错误不存储长字符串，大幅节省 Gas。
- **溢出检查 (unchecked)**：
    - 0.8.0 之后，编译器默认开启溢出检查。
    - **优化**：在确认数学逻辑绝对安全（如循环索引 `i++`）时，使用 `unchecked { ... }` 块包裹，可省去溢出检查的 Gas。

### 3. Foundry 核心测试技巧
- **事件测试 (vm.expectEmit)**：
    - `vm.expectEmit(checkTopic1, checkTopic2, checkTopic3, checkData)`
    - 前三个 `bool` 对应 `indexed` 参数（Topics），最后一个对应普通参数（Data）。
    - **用法**：先调用 `expectEmit`，紧接着触发 `emit` 预期事件，最后执行实际会触发事件的函数。
- **状态重置机制**：每一个 `test` 函数运行前，Foundry 都会重置 EVM 环境并重新执行一次 `setUp()`。
- **状态复用三策**：
    1.  **Modifier**：构建标准的状态前置条件。
    2.  **Internal Helper**：封装复杂的流程逻辑。
    3.  **Inheritance**：测试合约继承，实现基础环境共用。

### 4. 测试覆盖率 (forge coverage) 四大维度
1.  **Function**：函数入口是否被执行。
2.  **Line**：代码行是否被触达。
3.  **Statement**：拆分复合语句后的独立指令执行情况。
4.  **Branch**：逻辑分支生死线。必须同时覆盖 `true` 和 `false`（例如 `require` 的成功与失败）。

### 5. 工程化工具：Makefile
- **.PHONY 作用**：极其重要！声明伪目标，防止目录下有同名文件夹（如 `build/`）时导致命令失效。
- **变量聚合**：使用 `include .env` 配合环境变量，简化 `forge script` 部署命令。

### 6. 注释与提交规范
- **NatSpec (标准注释)**：
    - `@title`: 合约标题
    - `@author`: 作者
    - `@notice`: 给用户的普通说明
    - `@dev`: 给开发者的技术细节
    - `@param`: 参数解释
    - `@return`: 返回值解释
- **Conventional Commits (提交规范)**：
    - `feat:`: 新功能
    - `fix:`: 修复 Bug
    - `docs:`: 文档/笔记
    - `style:`: 格式调整（不改逻辑）
    - `refactor:`: 代码重构
    - `test:`: 测试代码
    - `chore:`: 琐事（配置/工具更新）

## 📅 3.29 | EVM 存储布局与继承深度解析

### 🧠 存储插槽 (Storage Slots) 基础
- **Slot 大小**：固定为 32 字节 (256 位)。
- **入座规则 (Packing)**：
    - **从右往左填**：第一个定义的变量占领 Slot 的低位（右侧）。
    - **紧凑打包**：如果相邻变量总和 <= 32 字节，会挤在同一个 Slot 中以节省 Gas。
    - **大端序表现**：EVM 虽是大端序，但打包逻辑让第一个变量看起来在最右边（低位）。
- **特殊变量**：`constant` 和 `immutable` 不占 Slot，直接编译进字节码。

### 🌳 继承中的存储布局 (Inheritance Storage)
- **长辈优先原则**：在继承链中，**父类合约的变量永远先占领起始 Slot**。
- **存储碰撞 (Storage Collision) 对策**：
    - **风险**：逻辑合约升级时，若父类变更导致 Slot 偏移，会覆盖原有数据。
    - **对策 A (Storage Gaps)**：在父合约末尾留空 `uint256[50] __gap;`，为未来预留空间。
    - **对策 B (Namespace Storage / ERC-7201)**：将变量存放在 `keccak256` 计算出的特定高位 Slot，彻底避开顺序排队的区域。
- **变量重名 (Shadowing)**：Solidity 0.6.0+ 严禁在继承体系中出现同名状态变量，编译器直接报错。

### ⚔️ 多重继承的优先级博弈
对于 `contract Vault is BaseA, BaseB, BaseC`:
- **存储 (Slots)**：**左侧优先 (Left-to-Right)**。`BaseA` 占起始 Slot。
- **逻辑 (Functions)**：**右侧优先 (Right-to-Left)**。最右侧的 `BaseC` 是“最派生”的，其重写逻辑优先级最高。
- **核心语法**：`virtual` (允许重写), `override` (执行重写), `super` (调用上一级逻辑)。

### 🛠️ 实用黑客命令：forge inspect
- `forge inspect <ContractName> storage`: 查看详细的存储布局、Slot 编号、Offset 偏移和字节数。
- `forge inspect <ContractName> abi`: 获取合约对外暴露的完整接口清单。
- `forge inspect <ContractName> bytecode`: 获取部署时的原始十六进制字节码。
- `forge inspect <ContractName> methodIdentifiers`: 获取所有函数的 4 字节 **Selector (选择器)** (如 transfer -> 0xa9059cbb)。

## 📅 3.30 | 代理模式、delegatecall 与存储安全对策

### 1. delegatecall：跨合约的“借尸还魂”
- **核心机制**：合约 A 委托调用合约 B，运行 B 的逻辑，但操作 A 的存储，且保留 A 的上下文。
- **三不变原则 (Context Preservation)**：
    - `address(this)` 不变：依然是 A 的地址。
    - `msg.sender` 不变：依然是最初发起交易的用户。
    - `msg.value` 不变：用户发送的 ETH 依然保留在 A 的上下文中。
- **call vs delegatecall 对比**：

| 维度 | call (外部调用) | delegatecall (委托调用) |
| :--- | :--- | :--- |
| **存储空间** | **被叫方 (B)** | **主叫方 (A)** |
| **address(this)** | B | A |
| **msg.sender** | A | 用户 (User) |

### 2. 代理模式 (Proxy Pattern) 架构
- **解耦设计**：
    - **Proxy 合约 (躯壳)**：负责持有资产、维护所有状态变量 (Storage)，地址永恒不变。
    - **Implementation 合约 (大脑)**：纯逻辑合约，可随时升级更换地址。
- **继承准则**：
    - **Proxy ❌ 不继承 Logic**：防止变量打架导致存储碰撞。
    - **Logic V2 ✅ 继承 Logic V1**：确保 V2 的变量自动排在 V1 之后，保证 Slot 绝对对齐。

## 📅 3.31 | 可升级合约的灵魂陷阱：Constructor 与 初始化

### 1. Constructor 的生命周期：卸磨杀驴
- **Creation Code (部署代码)**：包含 `constructor` 逻辑。部署时发送到链上，仅运行一次。
- **Runtime Code (运行时代码)**：合约地址最终保存的代码。不包含 `constructor`！
- **助推器比喻**：`constructor` 就像火箭的一级助推器，把合约推上轨道（初始化状态）后即被丢弃，不会留在轨道上。
- **对代理模式的影响**：Logic 合约的 `constructor` 代码不在链上，Proxy 无法通过 `delegatecall` 调用它。

### 2. 代理模式中的“初始化陷阱”
- **Logic 合约准则**：严禁在 `constructor` 里初始化需要在 Proxy 中持久化的状态。
- **正确对策：initialize() 函数**
    - 使用普通函数替代 `constructor`。
    - 必须配合“初始化保护器” (`initializer` 修饰器)，确保该函数在 Proxy 生命周期中只能被调用一次。
- **Proxy 合约准则**：避免在 Proxy 内部定义普通状态变量。如果定义了，极易与 Logic 产生 Slot 碰撞。

### 3. 存储安全进阶：EIP-1967 标准
- **核心思想**：不再使用 Slot 0 排队，而是计算一个极其偏远、随机的哈希槽位。
- **标准槽位示例**：`bytes32(uint256(keccak256("eip1967.proxy.implementation")) - 1)`。
- **效果**：变量“隐居”在几千亿光年外的 Slot 里，彻底杜绝碰撞。

- `vm.deal(addr, amt)`: 强行设置余额。
- `vm.prank(addr)`: 模拟发送者身份。
