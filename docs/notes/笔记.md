foundry中的作弊码如vm.expectEmit(true, true, false, true)前三个负责严格校验底层日志的Topic（带有indexed参数），最后一个严格校验底层日志的 Data所有非 indexed 参数的值，在其后可以紧跟emit假装触发一次完美事件（但不会写入日志），而是将这些具体的参数值存入虚拟机的预期比对库中。然后是具体函数执行最后将其底层的 Topics 和 Data 与存入的预期值进行字节级的对比

setUp() 的作用是定义一个基准初始状态。框架的实际调度逻辑是：在执行每一个单独的测试函数之前，EVM 虚拟机都会被重置，并重新执行一遍 setUp()。
如果要复用之前的状态有三种方法：1.使用 Modifier（修饰器）构建状态流（就正常的modifier定义方法相同）；2.封装 Internal 辅助函数 Helper Functions（需要动态传递参数或者返回某些生成的临时变量），在函数声明时加internal；3.测试合约继承

forge coverage有四层覆盖率：1.Function Coverage（函数覆盖率），只要这个函数的入口 JUMPDEST 指令被执行过一次，就算 100% 覆盖；2.Line Coverage（行覆盖率），只要这一行代码对应的任何一个 EVM 字节码被执行过，这一行就标绿；3.Statement Coverage（语句覆盖率），一行代码里可能有多个语句（比如 a = 1; b = 2;）。Foundry 会根据 AST 拆分语句，确保每一个独立的指令动作都被执行；4.Branch Coverage（分支覆盖率 —— 核心生死线），对于任何一个条件判断（if/else、require(condition, "error")），你必须在测试中让它既走过 true 的路径，又走过 false 的路径（即触发 revert）。

 NatSpec (Ethereum Natural Language Specification Format) 注释规范！
 标签    │ 作用               │ 什么时候用？                     │
  ├─────────┼────────────────────┼──────────────────────────────────┤
  │ @title  │ 合约的标题         │ 写在合约文件的最上方             │
  │ @author │ 作者的署名           │          │
  │ @notice │ 给普通用户看的说明 │ 解释这个函数是做什么的 │
  │ @dev    │ 给开发者看的硬核说明 │ 解释复杂的算法或安全性设计       │
  │ @param  │ 参数说明           │ 解释每一个传入变量的作用         │
  │ @return │ 返回值说明         │ 解释函数会吐出什么结果
